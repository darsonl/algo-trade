## Warnings

### WR-01: Tests use same data for both 1m and 1y SPY history calls — 1y result not independently validated

**File:** `tests/test_macro.py:136-158` and `tests/test_macro.py:175-191`

**Issue:** Both `test_fetch_macro_context_returns_correct_shape_on_success` and `test_fetch_macro_context_bearish_spy` set `mock_spy_ticker.history.side_effect = lambda period: pd.DataFrame({"Close": spy_closes})`. Because the lambda ignores the `period` argument, both the `"1mo"` and `"1y"` calls receive the same 20-element close series. As a result `spy_trend_1y` is computed from the same data as `spy_trend_1m`.

In the first test, the assertions only check `spy_trend_1y is not None` and that `spy_trend_1m` contains "Bullish" — the 1y label is never independently verified. In the second test, there is no assertion on `spy_trend_1y` at all. If the 1y slice logic were broken (e.g., wrong column name, inverted iloc index), these tests would still pass.

`test_fetch_macro_context_returns_spy_trend_1y` (line 222) correctly uses `side_effect=spy_history` with per-period dispatch and does independently assert the 1y label. The two earlier tests should be updated to match that pattern so that the 1m and 1y computations are independently exercised.

**Fix:** Replace the shared lambda with a per-period dispatch function in both tests:

```python
# test_fetch_macro_context_returns_correct_shape_on_success
spy_1m_closes = pd.Series([100.0] + [103.2] * 19)   # +3.2% for 1m
spy_1y_closes = pd.Series([100.0] + [112.5] * 251)   # +12.5% for 1y

def spy_history(period):
    if period == "1mo":
        return pd.DataFrame({"Close": spy_1m_closes})
    if period == "1y":
        return pd.DataFrame({"Close": spy_1y_closes})
    return pd.DataFrame({"Close": pd.Series([100.0, 100.0])})

mock_spy_ticker.history.side_effect = spy_history

# Then assert independently:
assert "Bullish" in result["spy_trend_1m"]
assert "Bullish" in result["spy_trend_1y"]   # different data, same direction is fine here
```

```python
# test_fetch_macro_context_bearish_spy
spy_1m_closes = pd.Series([100.0] + [98.6] * 19)    # -1.4% for 1m
spy_1y_closes = pd.Series([100.0] + [91.5] * 251)   # -8.5% for 1y

def spy_history(period):
    if period == "1mo":
        return pd.DataFrame({"Close": spy_1m_closes})
    if period == "1y":
        return pd.DataFrame({"Close": spy_1y_closes})
    return pd.DataFrame({"Close": pd.Series([100.0, 100.0])})

mock_spy_ticker.history.side_effect = spy_history

# Then assert both:
assert "Bearish" in result["spy_trend_1m"]
assert "Bearish" in result["spy_trend_1y"]
```

### WR-02: `fetch_macro_context` does not guard against an empty 1y history DataFrame

**File:** `screener/macro.py:94-97`

**Issue:** The 1-month history fetch is done first (line 88-91) and will raise `IndexError` if yfinance returns an empty DataFrame, which is caught by the outer `except Exception`. However, the 1-year fetch (lines 94-97) adds a second independent point of failure with the same `iloc[0]` / `iloc[-1]` access pattern. If yfinance returns a non-empty 1m DataFrame but an empty 1y DataFrame (e.g., a newly listed ETF, network partial failure, or a weekend/holiday edge case where yfinance returns 0 rows for an annual period), `spy_hist_1y["Close"].iloc[0]` raises `IndexError`, which the outer `except` catches and returns all-None. This is silent — the log message says "Failed to fetch macro context" without indicating whether it was the 1m or 1y fetch that failed, making debugging harder in production.

The risk is low (SPY has a multi-decade history) but the failure mode is silent and returns `None` for all three keys even though the 1m fetch succeeded.

**Fix:** Either validate both DataFrames before accessing, or at minimum log which period failed:

```python
spy_hist = spy.history(period="1mo")
if spy_hist.empty:
    raise ValueError("SPY 1m history returned empty DataFrame")
first_close = spy_hist["Close"].iloc[0]
last_close = spy_hist["Close"].iloc[-1]
spy_return = (last_close - first_close) / first_close

spy_hist_1y = spy.history(period="1y")
if spy_hist_1y.empty:
    raise ValueError("SPY 1y history returned empty DataFrame")
first_close_1y = spy_hist_1y["Close"].iloc[0]
last_close_1y = spy_hist_1y["Close"].iloc[-1]
spy_return_1y = (last_close_1y - first_close_1y) / first_close_1y
```

### IN-01: `format_spy_trend` docstring says "1-month return" but is now used for both 1m and 1y

**File:** `screener/macro.py:8-9`

**Issue:** The function docstring reads `"Format SPY 1-month return as a direction label with percentage."` and the `Args` description says `"one_month_return"`. After Phase 14.1 the function is called with both the 1-month and 1-year return values. The docstring is misleading for future maintainers.

**Fix:** Update the docstring to reflect the generic use:

```python
def format_spy_trend(period_return: float) -> str:
    """Format a SPY periodic return as a direction label with percentage.

    Args:
        period_return: Fractional return (e.g., 0.032 for +3.2%, -0.014 for -1.4%)

    Returns:
        "Bullish (+3.2%)" or "Bearish (-1.4%)"
    """
```
