<objective>
Implement ETF-09: ETFs with an expense ratio above a configurable threshold are visibly flagged in their Discord embed.

Purpose: Currently the ETF embed shows the raw expense ratio value. Operators want high-cost ETFs visually marked so they can factor cost into their Approve/Reject decision without cross-referencing external data.

Output:
- `Config.etf_max_expense_ratio: float` field defaulting to 0.005 (0.5%) from `ETF_MAX_EXPENSE_RATIO` env var
- New `etf_max_expense_ratio: float | None = None` parameter on `build_etf_recommendation_embed` that, when provided and `expense_ratio > threshold`, renders the Expense Ratio field value as `"⚠️ 0.0075 (High)"` instead of `"0.0075"`
- `send_etf_recommendation` on TradingBot threads the new kwarg to the embed builder
- `run_scan_etf` passes `config.etf_max_expense_ratio` through to `send_etf_recommendation`
- Unit tests for all four cases: above / at / below / None-expense-ratio / None-threshold
- `.env.example` entry for `ETF_MAX_EXPENSE_RATIO`

All flag logic is display-only per D-05 — no skip/filter.
</objective>

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-04 | Information Disclosure | Discord embed shows expense ratio | accept | Already public information; no PII, no secrets |

| T-12-06 | Spoofing | Fake ⚠️ character causing embed rendering issues | mitigate | Hardcoded UTF-8 literal; Discord renders standard emoji; no user input reaches this field |

## Residual Risk
Low — display-only change on an existing embed. No new trust boundary crossed; no code path leads to trade execution based on the flag. The strict `>` comparison cannot cause false negatives at the boundary because at-threshold ETFs are intentionally unflagged per D-07.
